# 《College》程序架构与运行状态

> **文档定位**：记录当前 Godot 工程的真实程序结构、运行入口与接线状态。  
> **判断原则**：以 `project.godot` 与当前 `scripts/core/*`、`scenes/*` 代码为准；历史文档只作参考。  
> **最后核对**：2026-04-07

---

## 1. 当前工程结论

### 1.1 当前真正运行入口

当前项目的运行入口是：

- `project.godot` → `run/main_scene="res://scenes/test_scene.tscn"`

这意味着当前工程默认启动的是：

- `scenes/test_scene.tscn`
- 其脚本为 `scripts/debug/test_runner.gd`

**结论**：当前工程主链是“21 天自动测试 / 数值验证链路”，不是正式玩家 UI 主场景。

### 1.2 当前程序状态一句话判断

当前 `College` 已经具备：

- 可运行的核心循环系统
- 可验证的 21 天测试链路
- 一套已搭起但尚未完全重新接线的 UI 骨架
- 一批历史文档与旧接口，需要统一降级为参考

## 2. Autoload 单例架构

`project.godot` 当前注册了以下 Autoload：

| 单例 | 路径 | 当前职责 |
|---|---|---|
| ThemeColors | `scripts/core/theme_colors.gd` | 主题色配置 |
| GameManager | `scripts/core/game_manager.gd` | 全局阶段、天数、时段索引 |
| AttributeSystem | `scripts/core/attribute_system.gd` | 属性与资源数据 |
| RelationshipSystem | `scripts/core/relationship_system.gd` | NPC 好感与衰减 |
| CardSystem | `scripts/core/card_system.gd` | 卡牌定义、解锁与发牌 |
| EventSystem | `scripts/core/event_system.gd` | 事件队列、条件判断、优先级仲裁 |
| DayManager | `scripts/core/day_manager.gd` | 日循环编排、卡牌执行、结算输出 |

### 2.1 当前职责边界

#### GameManager
负责：
- 当前阶段 `Phase`
- 当前天数 `current_day`
- 当前时段索引 `current_period_index`
- Demo 结束判定
- 军训日期判定

不负责：
- 发牌
- 事件队列生成
- 单日执行编排
- UI 状态快照拼装

**判断**：当前 `GameManager` 是“全局状态持有者”，不是旧版意义上的大一统流程控制器。

#### DayManager
负责：
- 开始一天时的系统编排
- 每时段 AP 初始化
- 强制事件检查
- 军训锁定时段处理
- 发牌入口
- 卡牌执行与效果落地
- 当日总结数据输出
- 推进到下一天

**判断**：当前 `DayManager` 才是核心循环的真正主控。

## 3. 当前主程序链路

### 3.1 启动后流程

当前默认流程为：

1. 启动项目
2. 进入 `scenes/test_scene.tscn`
3. 执行 `scripts/debug/test_runner.gd`
4. `test_runner.gd` 调用 `GameManager.reset()`
5. 手动注册 Demo NPC 初始好感
6. 调用 `DayManager.start_day()`
7. 自动跑完整个 21 天循环

### 3.2 测试链路的作用

`test_runner.gd` 当前承担的是：

- 核心系统是否能完整跑通
- 军训期精力压力是否成立
- 手牌中的保底卡 / 随机卡差异是否成立
- 顾遥好感在 Day 20 是否逼近目标区间

**结论**：当前主链更接近“系统验算场景”，而不是正式 Demo 演出入口。

## 4. 核心系统状态

### 4.1 GameManager

当前已实现：
- `TITLE / SCHEDULING / EXECUTING / EVENT / DAY_SUMMARY / DEMO_END` 六阶段
- 时段顺序：上午 / 下午 / 晚间
- `MAX_DAY = 21`
- Day 3-9 军训日判定
- `advance_period()` / `advance_day()`

当前特征：
- `limited_days` 已保留为空数组
- `is_period_locked()` 仍检查 `limited_days`
- 军训锁定的真正执行逻辑在 `DayManager`

### 4.2 AttributeSystem

当前已实现：
- 四维属性：学识 / 口才 / 体魄 / 见识
- 三项资源：精力 / 金钱 / 心情
- 每日精力回复为 **7**
- 低精力阈值为 **精力 ≤ 2**
- `get_roll_modifier()` 已提供骰子修正计算

当前判断：
- 属性与资源系统已稳定可用
- 骰子相关工具函数已存在，但完整骰子玩法仍未挂入正式主链

### 4.3 RelationshipSystem

当前已实现：
- 好感范围 0-100
- 阈值：20 / 40 / 60 / 80
- 衰减冷却：5 天
- 注册免疫期：7 天
- 衰减值：-2
- 正向互动会清零未互动天数

当前判断：
- 关系系统已可支持 Demo 数值循环
- NPC 注册目前仍由测试链路显式完成，不是独立存档系统

### 4.4 CardSystem

当前已实现：
- 卡牌类 `ActionCard`
- 手牌数 `HAND_SIZE = 5`
- 常驻保底卡
- 课程保底卡
- 随机池补足
- 天数 / 单 NPC 好感 / 任意 NPC 好感解锁
- Day 2 推荐卡机制接口
- Day 10+ 课程表保底机制

当前判断：
- 这是当前 Demo 核心可玩性的主要承载系统之一
- 当前实现已经明确是“发牌式行动池”，不是固定菜单式行动列表

### 4.5 EventSystem

当前已实现：
- 固定事件 / 条件事件 / 随机事件
- `RANDOM_EVENT_CHANCE = 0.3`
- 每日队列构建
- 固定事件优先仲裁
- 非固定事件顺延标记 `deferred`
- `attach_to` 机制
- 事件效果可直接写入属性 / 资源 / 好感

当前判断：
- 事件系统已经具备基础调度能力
- 但事件演出层与 UI 的数据适配仍未彻底做完

### 4.6 DayManager

当前已实现：
- 一日开始时统一调度
- 每时段 2 AP
- 强制事件优先占用时段
- 军训锁定时段直接执行军训效果
- 卡牌执行后结算弱事件
- 输出日结算 `summary`

当前判断：
- 当前项目最关键的程序文件是 `scripts/core/day_manager.gd`
- 其输出数据结构已经成为新文档的事实基准

## 5. 当前运行状态与 UI 的关系

### 5.1 现存 UI 场景链

工程里仍然存在以下 UI 场景：

- `scenes/main.tscn`
- `scenes/ui/ui_test_root.tscn`
- `scenes/ui/ui_test_v2.tscn`

它们的定位并不相同。

### 5.2 `scenes/main.tscn`

当前只是一个薄包装场景：

- `scenes/main.tscn` 实例化 `scenes/ui/ui_test_root.tscn`

但它**不是当前运行入口**。

### 5.3 `ui_test_root.tscn` / `scripts/ui/ui_test_root.gd`

这一套可以看作 **旧版 UI 主控链**：

- 场景层级完整：TodayView / StoryLayer / DiceLayer / PhoneLayer / DaySummaryLayer
- 但脚本接口依赖旧版 core API

当前失配点包括：
- 调用了不存在的 `GameManager.reset_run()`
- 监听了当前 `DayManager` 中不存在的 `event_started`、`action_resolved`
- 监听了当前 `GameManager` 中不存在的 `state_changed`
- 调用了不存在的 `DayManager.select_card()`
- 调用了旧命名的 `DayManager.continue_next_day()`
- 读取了当前 `GameManager` 中不存在的 `stats`、`relationships`、`daily_points_remaining`、`period_points_remaining`

**结论**：
`ui_test_root.gd` 代表的是旧接口时代的 UI 主控，不应再描述为“已接通的正式主链”。

### 5.4 `ui_test_v2.tscn` / `ui_test_v2.gd`

这一套更像 **并行 UI 原型 / 展示场景**：

- 通过按钮分别测试 TodayView、StoryLayer、DiceLayer、PhoneLayer、DaySummaryLayer
- 输入数据主要是手写假数据
- 重点在界面样式、层级结构与演出方向验证

**结论**：
`ui_test_v2` 不是当前核心循环运行入口，而是 UI 单层展示与验证场。
