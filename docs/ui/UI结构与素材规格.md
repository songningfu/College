# 《College》UI 结构与素材规格

> **文档定位**：统一当前 UI 分层结构、场景状态、接线现状与素材出图规格。  
> **核心原则**：区分“已存在的界面骨架”“当前默认运行链”“尚未重新接通的接口”。  
> **最后核对**：2026-04-07

---

## 1. 当前 UI 结论

### 1.1 当前不要再误写成的说法

以下表述当前都不准确：

- “UI 已全部接通”
- “主入口就是 `scenes/main.tscn`”
- “Today / Story / Dice / Phone / Summary 已完整联动当前 core”

### 1.2 当前准确说法

当前 UI 状态应表述为：

- 已有一套明确的层级式 UI 目标结构
- 工程内同时存在旧版主控链与新版展示原型
- 当前默认运行入口仍是测试场景，不是正式 UI 主场景
- UI 与 core 的整链接线还没有完全完成

---

## 2. 当前 UI 目标结构

《College》当前 UI 方向已经比较清楚：

**顶部状态栏 + 中景舞台区 + 时段信息区 + 底部手牌区 + 独立覆盖层**

其中覆盖层包括：

- StoryLayer
- DiceLayer
- PhoneLayer
- DaySummaryLayer

这套结构服务的是四层体验：

- 计划层
- 执行层
- 演绎层
- 结算层

---

## 3. 当前场景结构

### 3.1 现存主要 UI 场景

工程内当前与 UI 直接相关的主场景有三类：

| 场景 | 当前定位 |
|---|---|
| `scenes/main.tscn` | 旧 UI 包装入口 |
| `scenes/ui/ui_test_root.tscn` | 旧版整合型 UI Root |
| `scenes/ui/ui_test_v2.tscn` | 新版分层 UI 展示 / 测试场景 |

### 3.2 `scenes/main.tscn`

当前内容很薄，只是实例化：

- `scenes/ui/ui_test_root.tscn`

因此它更像历史上的主入口包装场景。

### 3.3 `scenes/ui/ui_test_root.tscn`

这一套场景结构已经把五个核心层组装在一起：

- TodayView
- StoryLayer
- DiceLayer
- PhoneLayer
- DaySummaryLayer

它代表的是：

**旧版“一套场景内集成各层”的 UI 方案。**

### 3.4 `scenes/ui/ui_test_v2.tscn`

这一套则是新版测试舞台：

- 使用 v2 版本的 Today / Story / Dice / Phone / Summary 场景
- 带测试按钮
- 手动灌入示例数据

它代表的是：

**更偏演出与排版验证的新原型。**

---

## 4. 旧版整合链的现状

### 4.1 `scripts/ui/ui_test_root.gd` 的定位

这个脚本当前不能视为“已可直接驱动正式 Demo 的主控”。

它更准确的定位是：

- 早期 UI 主控脚本
- 面向旧版 core 接口编写
- 场景结构仍有参考价值
- 但接口已明显老化

### 4.2 当前已确认的失配

`ui_test_root.gd` 里存在多处与现行 core 不匹配的问题，包括：

- 调用 `GameManager.reset_run()`，但当前 `GameManager` 只有 `reset()`
- 监听 `DayManager.event_started`，但当前为 `event_triggered`
- 监听 `DayManager.action_resolved`，但当前为 `execution_finished`
- 监听 `GameManager.state_changed`，但当前为 `phase_changed` / `day_changed`
- 调用 `DayManager.select_card()`，但当前有效入口是 `play_card()`
- 调用 `DayManager.continue_next_day()`，但当前有效入口是 `continue_to_next_day()`
- 依赖 `GameManager.stats`、`relationships` 等旧快照字段，但当前这些状态分散在各系统中

### 4.3 结论

`ui_test_root.tscn` 的**层级结构可参考**，但 `ui_test_root.gd` 的**主控接口不能直接当现行主链使用**。

---

## 5. v2 UI 原型链的现状

### 5.1 `ui_test_v2.gd` 的使用方式

当前 `ui_test_v2.gd` 的主要功能是：

- 点击按钮测试某一层
- 手写一份示例数据
- 直接调用对应 layer 的显示接口

它能验证：

- 布局是否成立
- 信息密度是否合适
- 某一层单独展示是否顺眼

它不能证明：

- 已与真实 core 数据接通
- 已与当前事件对象结构适配
- 已经是正式运行入口

### 5.2 结论

v2 是**视觉与交互方向原型**，不是当前程序主链。

## 6. 五个核心 UI Layer 的职责

### 6.1 TodayView

当前职责应是：

- 展示当天 / 当前时段
- 展示资源与属性摘要
- 展示本时段 AP 状态
- 承载底部可选行动卡
- 作为主界面计划层入口

它对应的是：

- **计划层**
- 一部分**轻执行反馈**

### 6.2 StoryLayer

当前职责应是：

- 呈现剧情事件标题
- 承载正文描述
- 展示选项
- 展示事件结果文本

它对应的是：

- **演绎层**

当前注意点：
- 现有 `story_layer.gd` 接收的是 `Dictionary`
- 当前 `DayManager` 发出的是 `RefCounted event`
- 中间仍缺一层事件对象到 UI 数据的适配

### 6.3 DiceLayer

当前职责应是：

- 独立展示某次检定
- 强化悬念与反馈
- 承载成功 / 失败结果展示

当前状态：
- 独立层原型存在
- 但核心系统只提供 `AttributeSystem.get_roll_modifier()` 这类基础支持
- 仍未形成稳定接回主链的完整判定流程

### 6.4 PhoneLayer

当前职责应是：

- 承载消息线程
- 展示联系人与未读状态
- 作为关系延续入口

当前状态：
- 原型界面已存在
- v2 展示中使用的是手写联系人数据
- 当前尚不是正式连接 Relationship / Event / Dialogue 的真实系统

### 6.5 DaySummaryLayer

当前职责应是：

- 展示当天结束后的变化
- 呈现属性 / 资源 / 关系回流
- 提供下一天推进入口

当前状态：
- 结算层界面原型已存在
- 但旧版 `day_summary_layer.gd` 仍读取旧结构 `summary.stats` / `summary.relationships`
- 与当前 `DayManager` 发出的 `final_attributes` / `final_resources` / `final_relations` 不一致

## 7. UI 与核心循环的当前关系

### 7.1 当前真正跑通的链

当前真正稳定跑通的是：

- core systems
- test runner
- 21 天自动测试与日志输出

### 7.2 当前 UI 的实际定位

UI 当前更准确的说法是：

- 有明确结构
- 有旧版整合稿
- 有新版展示稿
- 但还没有形成“可直接替代测试场景的正式运行 UI 主链”

### 7.3 当前最关键的接线缺口

1. `DayManager` 信号名与 UI 主控期待值不一致
2. `GameManager` 不再提供旧版快照式字段
3. StoryLayer 需要事件对象适配
4. DaySummaryLayer 需要 summary 字段适配
5. PhoneLayer 仍缺真实数据源

---

## 8. 当前信息架构口径

后续所有 UI 相关文档，统一按下面这套信息架构描述：

### 8.1 顶部状态栏

建议承载：
- 精力
- 心情
- 金钱
- 手机入口
- 电脑入口（可先保留占位）

### 8.2 中景舞台区

建议承载：
- 校园 / 宿舍 / 教室 / 操场等背景
- 人物站位
- 一句环境提示或轻反馈
- 轻剧情氛围

中景不是大段说明面板。

### 8.3 时段信息区

建议承载：
- 当前是上午 / 下午 / 晚间
- 本时段剩余 AP
- 当前状态提示

### 8.4 底部手牌区

建议承载：
- 本时段可选卡牌
- 卡牌取舍
- 确认执行

这仍然是当前主交互重心。

### 8.5 覆盖层

当进入强剧情、骰子、手机或日结算时，再切入对应覆盖层。

## 10. 当前素材风格要求

当前 Demo 的视觉风格建议保持：

- 校园青春感
- 半真实、半戏剧
- 生活感和记忆感优先
- 不做过度夸张、纯炫技式二次元表现
- 让画面服务关系、事件与氛围

### 10.1 场景图

优先级最高的是：

- 宿舍白天 / 夜晚
- 校门口
- 教学楼外
- 食堂
- 军训操场
- 校园夜路
- 社团活动点

### 10.2 卡牌图

要求：

- 一眼能认出动作类型
- 单主体清晰
- 细节不要过碎
- 缩小后仍能读图

### 10.3 头像与联系人图

要求：

- 小尺寸仍有辨识度
- 脸部特征优先
- 不同角色之间轮廓差异清晰

## 11. UI 文档表述规范

今后提到 UI 状态时，统一使用下面三类词：

### 可写“已实现”的
- 分层目标结构
- 多个 layer 场景原型
- v2 手动展示场景
- 旧版整合 root 场景本身存在

### 只能写“已有骨架 / 尚未完全接通”的
- 主界面与 core 的实时联动
- StoryLayer 与事件系统的数据流
- DaySummaryLayer 与当前 summary 的适配
- PhoneLayer 的真实关系推进链路
- DiceLayer 的正式主链接入

### 必须避免的
- “UI 主流程已经做完”
- “所有层都已接通当前程序”
- “正式玩家入口已经换到 main.tscn”

---

## 12. 当前 UI 侧下一步重点

如果后续开始正式整 UI，建议顺序是：

1. 先决定保留 `ui_test_root` 还是以 `ui_test_v2` 为新主控基础
2. 为 TodayView 建立当前 core 快照拼装层
3. 把 `event_triggered` 的 `GameEvent` 转成 StoryLayer 可消费字典
4. 让 DaySummaryLayer 改读 `changes` / `final_*` 结构
5. 再把 PhoneLayer 从假数据切到真实数据

这样能在不破坏现有核心循环验证链的前提下，逐步把 UI 接回来。
